## 方法

- 每个里程碑都有明确的验收闸门；未经验证的结果不进入下一步。
- 每个重要技术选择都做了调研（2026 最佳实践），并连同取舍记入 [DECISIONS.md](./DECISIONS.md)，使每个决策都可被 defend。

## AI 犯的错，以及我如何抓到

**速率滑杆并没有真正驱动 feed。** 自动化数字全部通过——120fps、`Applied/s = 800`——但一张截图显示滑杆写着 "200/s"，而 feed 实际跑在 800：AI 把滑杆接成了一个装饰性控件，而非速率的真相源。断言抓不到它（滑杆和 feed 各自内部自洽，只是彼此不一致），一次肉眼审查抓到了。修复：客户端现在在连接时与每次变更时都把滑杆的速率断言给服务端，两者不再漂移。

**变更高亮在不该闪时闪了。** 初版用 `rev` 作 key 重挂载高亮覆盖层，导致每次挂载都触发——于是初始加载时每行都闪、每次滚入视图又闪一次。我在运行前通过推敲"挂载 vs 原地变更"的语义抓到，加了 mount 守卫使只有真正的原地更新才闪。评审时又发现一个相关情形——筛选列表重排复用行槽位会闪*错*货运——通过把虚拟器的 key 从 index 换成 `reference` 修复。

## 交付前评审（多代理）

打包前，我用三个并行评审子代理（正确性/bug、React 渲染性能、清晰度/健壮性）跑了一轮评审，再对发现分级处理。高价值、低风险项当场修复并复验；边角情况记为"下一步"。值得一提的修复：

- **WebSocket 带退避的自动重连**——客户端此前会永久卡在 `disconnected`；现在能从启动竞态和服务端重启中恢复（已验证：杀掉 feed → `disconnected` → 重启 → 自动重连 → 数据恢复）。
- **虚拟器按 reference 的 `getItemKey`**——修复了筛选列表重排复用行槽位时的误闪，并在筛选 churn 下保持 `React.memo` 有效。
- **服务端崩溃守卫**——`null` 客户端消息不再拖垮 feed；空数据集与缺失 CSV 的路径都已处理；`SIGINT` 关闭不再因存活 socket 而卡住。
- **可移植性**——把 `import.meta.dirname`（需 Node ≥20.11）换成 `fileURLToPath`；移除两个未使用的 devDependency；把误导性的 HUD "Buffer" 指标改名为 "Batch"。

## 生成 vs. 手写 / 主导

- **AI 生成、工程师评审**：脚手架、feed 引擎、store、虚拟表、HUD、CSS 均由 AI 起草，随后逐行评审与修正（上面几条纠错正是从这一评审中得出）。
- **工程师主导**：架构与每一个技术选型（传输、状态管理、虚拟化、批处理、单包布局）都是带着有据可查的取舍刻意决定的（DECISIONS.md），且每个里程碑都在推进前对照验收闸门做了验证。
