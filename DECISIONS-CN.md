# 决策与取舍（ADR-lite）

> 本文为 [`DECISIONS.md`](./DECISIONS.md) 的中文版。技术标识符与工具名保留原样。

塑造本次构建的决策流水账——架构答辩的素材。每条：背景 → 选项 → 决策 → 理由 → 放弃了什么。

---

### D1 — Feed 传输：独立 Node WebSocket 服务

- **背景**：题目给了一份静态 CSV，要求我们自行设计实时 feed、传输与工具链——"be ready to defend it"。
- **选项**：(a) 独立 WS 服务；(b) 浏览器内 Web Worker 生成器；(c) SSE。
- **决策**：独立 Node `ws` 服务（`server/`）。
- **理由**：最接近真实生产 feed——真正的网络边界让我们能推理传输、批处理与重连，也让"10× 行 / 多看板 / 离线网点"的扩展讨论落到实处。本地 Node 进程**不是**"外部服务"（那指 Redis / 云 broker），因此"单命令 + 无外部服务"规则仍成立。
- **放弃**：进程内生成器的极致简单；我们现在要管第二个进程（交给 `concurrently`）。

### D2 — 状态管理：Zustand + 按行 selector + 结构共享

- **背景**：1 万行、每秒数百次更新。失效模式（来自调研）是朴素 store 在每次更新时重渲染整个列表。
- **选项**：(a) 自建外部 store + `useSyncExternalStore`；(b) Zustand；(c) Valtio；(d) Jotai。
- **决策**：Zustand，每行通过 selector 只订阅**自己那条**，flush 时产出一个复用未变行引用的新 map（结构共享）。
- **理由**：成熟、API 极简；底层本就基于 `useSyncExternalStore`。结构共享意味着未变行的 selector 返回同一引用（`Object.is`）→ 不重渲染；只有变更行重渲染。这化解了"整表重渲染"陷阱。
- **放弃**：手写 store 原语的教学价值；Valtio 的自动 proxy 追踪（10k 下更难 defend）与 Jotai 的每行一 atom（此处更重）。

### D3 — 虚拟化：TanStack Virtual

- **选项**：(a) TanStack Virtual；(b) react-window；(c) react-virtuoso。
- **决策**：TanStack Virtual。
- **理由**：活跃维护（2026）、headless/完全可控，与自定义行渲染器和性能叙事契合。1 万行下三者都能上 60fps；瓶颈在行渲染器而非库本身——所以我们优化渲染器。
- **放弃**：react-window 更小的 API 面（但它已停更、且仅支持定高）。

### D4 — 工程结构：单包，文件夹 `web/ server/ shared/`，纯 npm

- **背景**：题目把"照文档命令能跑起来"的权重放在任何单一功能之上——跑不起来，申请就无法推进。
- **选项**：(a) pnpm/turbo monorepo；(b) npm workspaces；(c) 单包 + 文件夹。
- **决策**：单个 `package.json`、源码按文件夹划分、纯 npm、用 `concurrently` 实现单命令。提交的 `package-lock.json` 锁死依赖树。
- **理由**：只需 Node——一次 `npm install`、一条 `npm run dev`、零额外工具链步骤（无需 `corepack enable`）。面试官冷启动时失败点最少。模块边界仍靠文件夹 + `shared/` 契约保持清晰。
- **放弃**：真正 monorepo 的按包依赖隔离与缓存粒度——在 ARCHITECTURE.md 里记为团队规模化的"下一步"。
- **何时我才会上真正的 monorepo**（诚实的边界，而非"monorepo 不好"）：一旦出现共享代码的*独立消费者*——例如"多看板"扩展场景中，多个前端应用共享 `@tv/ui` 组件库和 `@tv/protocol` 包。那时 pnpm workspaces + Turborepo 的构建缓存才回本。在 3 个文件夹、1 个可运行应用、0 个发布包时，它们不回本。对的工具，对的规模。

### D5 — 更新批处理：ref 缓冲 + requestAnimationFrame flush

- **背景**：每秒数百条 delta 不能翻译成每秒数百次 React 渲染。
- **决策**：把到达的 delta 缓冲进 ref；每个动画帧 flush 一次，同 reference 的多次更新合并取最新，跳过值未变的 no-op。
- **理由**：把任意到达速率塌缩成每帧最多 1 次提交（~16.7ms），与浏览器绘制周期对齐；rAF 在后台标签页还会自动降频。
- **放弃**：相比同步更新，多了略微的更新延迟（≤1 帧）——不可感知，是换取流畅的正确取舍。

### D6 — Feed 生成器：服务端权威态、按 tick 批量 delta、生命周期流转

- **背景**：feed 必须推送"峰值每秒数百条、速率可配"，且要可 defend。
- **决策**：
  1. 服务端持有唯一权威态（`Map<ref, Shipment>`）；客户端连接时拿到*当前*态的快照，因此多客户端/重连都保持一致。
  2. 生成器每 50ms 醒来，把 `rate × 0.05` 条更新作为**一条批量 delta** 发出，而非每条更新一个 WS 帧——帧数少得多，吞吐相同。
  3. 更新遵循真实生命周期（`created → picked_up → in_transit → delivered/failed → 回收`）而非纯噪声，故每条更新都是真实变更，看板像真实运营看板一样翻动。
- **理由**：服务端权威态是诚实的模型，并让"多看板 / 重连 / 离线"的扩展叙事落到实处。批处理把吞吐与帧数解耦。生命周期翻动让高亮 + 筛选成员逻辑在 demo 中得到真实检验。
- **放弃**：每客户端独立 feed（所有客户端共享一个全局速率——记为下一步）；更重的流式 CSV 解析器（对干净、扁平的数据集不需要）。
- **已验证**：`FEED_RATE=300` → 实测约 295 条更新/秒；快照 = 10,000 行。

### D7 — 变更高亮：以 rev 为 key 的淡出覆盖层 + mount 守卫

- **背景**："行原地更新，变更时短暂高亮"——但必须*只*在原地变更时闪，加载或滚入视图时都不闪。
- **决策**：每行带一个 `rev` 计数器（每次更新 +1）。行渲染一个 `<span key={rev}>` 覆盖层，播放一次性 CSS 淡出；改变 `key` 即重播。mount 守卫（`mounted` ref + `rev > 0`）抑制首次挂载时的闪烁，因此初始 1 万行加载与滚入都不触发。
- **理由**：声明式且廉价（只有可见且真正变更的行会重挂一个极小的 span）；淡出是纯 CSS（合成器层），并被 `prefers-reduced-motion` 关闭。
- **放弃**：JS 驱动的动画重启（reflow hack）——每帧更多强制布局。
- **已验证（M4，CDP + 注入式 FPS 计）**：在 **800 条更新/秒**下，静止与持续滚动均持续 120fps（120Hz 显示器）；`Applied/s` 跟随 feed、缓冲稳定（无背压）；滑杆端到端驱动服务端速率（200 → 1200）。

### D8 — churn 下筛选：只重算 churn 能改变的部分

- **背景**：状态筛选 + 文本搜索必须在数据翻动时既正确又响应。
- **关键洞察**：delta 改 `status` 和 `last_update`，但绝不改 `reference`/`customer`。所以**搜索**成员在 churn 下不变（其 key 从不改变）；只有**状态筛选**的成员会随行流转而变。这把难点精确收窄。
- **决策**：
  - `filteredOrder` 是表格渲染的派生视图。它在用户操作时重算（切换状态 / 输入搜索），以及在每次 delta flush 时**仅当状态筛选激活**才重算。纯搜索或无筛选视图在 churn 下从不重算。
  - 搜索防抖（150ms）→ 每次击键不做全表扫描。
  - 每行在加载时预计算一个小写 `searchKey` → 零分配子串匹配。
  - 实时结果计数被隔离进独立组件，故 churn 只重渲染计数，而非搜索框或 chip。
- **理由**：正确性有保证（激活筛选的路径每次 flush 都重算成员），同时消除无谓工作（常见情形不重算）。一次全 10k 扫描是亚毫秒级的，且只在真正可能影响结果时才跑。
- **已验证（M5，CDP churn 下）**：搜索 / 状态 / 组合均正确；状态筛选激活、**约 2000 条更新/秒**下，10 轮滚动 churn 显示 **0** 行违反筛选，持续 120fps。

### D9 — 测试范围：只测棘手部分，刻意不做宽泛 UI 覆盖

- **背景**：题目要求"针对真正棘手的部分测试——如批量更新或 churn 下筛选"，并明确说"无需宽泛的 UI 快照覆盖"。
- **决策**：26 个聚焦纯逻辑的单测，无 UI 组件/快照测试：
  - 客户端：`DeltaBatcher`（rAF 批处理/合并）、`boardStore`（结构共享、合并、筛选、churn 下筛选）；
  - 服务端：`FeedEngine`（`rate→count` 换算、生命周期无 no-op 保证、clamp、start/stop）、`parseShipments`（对格式错/未知 status 行的守卫、CRLF）。
- **理由**：这契合题目意图与 timebox。宽泛 UI 覆盖会违背题目、收益甚微（虚拟器是库的职责——改用 CDP 集成脚本验证）、且读作镀金。为让 batcher 与 CSV 解析可单测，把它们的逻辑从副作用外壳（`requestAnimationFrame` / `fs`）中抽出，藏在可注入的调度器与纯函数 `parseShipments(text)` 之后——一次由可测性驱动的重构。
- **下一步会加**：一个 Playwright 冒烟测试，在真实浏览器中断言负载下 FPS 与 churn 下筛选（CDP 脚本已证明这一点；Playwright 能让它在 CI 中跑）。
