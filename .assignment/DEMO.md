# Demo 演示口播稿（现场 ~5 分钟）

> 私用（放 `.assignment/`，不交付）。这是一份**能照着念**的口语稿：左边是操作动作，正文是可以直接说出口的话。语气自然点、别背书感。核心节奏：先证明"能跑起来" → 再证明"峰值下不卡" → 最后证明"边翻动边筛选也不出错"。

---

## 演示前（提前 2 分钟做好，别当场装）

```bash
cd live-ops-board
npm install        # 首次；已装可跳过
npm run dev        # 一条命令同起 feed + web
```
- 等终端出现 `feed ... listening on ws://localhost:8080` 和 `VITE ... ready`。
- 浏览器打开 **http://localhost:5173**，确认右上角是绿色 `connected`。
- 旁边留一个终端窗口（等下要展示"就一条命令"）。
- 想一上来就有明显流动，可以用 `FEED_RATE=800 npm run dev`。
- 万一端口被占：`lsof -ti tcp:8080 tcp:5173 | xargs kill -9` 再起。

## ① 开场（约 30 秒）——先讲"能跑起来"

**〔动作〕** 指一下终端，再指页面右上角。

> "在开始之前先说一句——这整个东西，前端加后端的实时 feed，我就用**一条命令** `npm run dev` 起来了，除了 Node 没有任何别的依赖，不需要数据库、不需要任何外部服务。
>
> 大家现在看到的，是一张**一万行**货运的实时运营看板，右上角显示 10,000 shipments、已经连上了。
>
> 我先把这道题的本质讲清楚：难点其实不在数据——数据是很干净的。真正难的是，**当每秒有几百条状态更新涌进来的时候，这个一万行的表格还能不能保持 60、甚至 120 帧不掉**。接下来我就顺着这条线演示。"

## ② 虚拟化（约 40 秒）——"一万行是怎么扛住的"

**〔动作〕** 鼠标快速上下滚动列表。

> "先滚一下——大家可以看到，一万行滚起来是很顺的，没有任何卡顿。"

**〔动作〕** 打开 DevTools（F12）→ Console，敲：
```js
document.querySelectorAll('.row').length
```
> "这里有个关键：我数一下现在页面上真实存在的行——大家看，**只有三十来个**，不是一万个。也就是说不管数据有多少，DOM 里始终只渲染可视区那几十行。这是虚拟化在做的事，滚动成本跟数据量完全无关。"

**〔可选〕** 再敲 `document.querySelector('.board__scroll').scrollHeight` → 400000。
> "这个 40 万，是一万行乘每行 40 像素撑出来的总高度，所以滚动条的比例是真的，但渲染的永远只是那一小片。"

## ③ 实时更新 + 峰值压测（约 90 秒）——全场高潮

**〔动作〕** 指 HUD 那一排读数（FPS / Applied/s / Batch / Feed lag / Visible）。

> "右上这排是我加的一个性能仪表盘，等下拉高负载的时候，大家可以盯着这几个数看。我快速说一下这五个数：
>
> **FPS** 是每秒渲染多少帧，是最关键的稳定性指标——稳在 60 或 120 就是不掉帧；
> 
> **Applied/s** 是每秒真正应用了多少条更新，它应该跟我拉的速率对得上，说明客户端消费得过来；
> 
> **Batch** 是每一帧合并处理了多少条更新，正常应该是稳定的、不会一直往上涨，一涨就说明开始堆积、跟不上了；
> 
> **Feed lag** 是从服务端发出到客户端处理的延迟，本地基本是 0 毫秒；
> 
> **Visible** 是当前真实渲染在页面上的行数——你会看到它永远只有几十行，不管数据有一万行还是十万行。"

**〔动作〕** 把 **FEED RATE** 滑杆从当前值一路拖到 **2000/s**。

> "我现在把 feed 的速率往上拉……拉到每秒两千条更新。
>
> 大家看三个地方：第一，**FPS 稳稳地在 120，一帧都没掉**；第二，Applied/s 精确地跟到了两千，说明客户端完全消费得过来；第三个 Batch 这个数很稳、没有一直往上涨——这说明**没有背压**，数据没有在客户端堆积。
>
> 秘诀其实就一句话：WebSocket 每来一条更新我**不是**立刻去 setState 的，而是先塞进一个缓冲区，然后**每一帧只提交一次**。所以哪怕每秒来两千条、五千条，React 每帧最多也只渲染一次。"

**〔动作〕** 指着某几行正在闪的高亮。

> "另外注意看这些行在变的时候会**闪一下**——而且只有可视区里真正变了的那几行会重新渲染，屏幕外面变的行只更新数据、根本不碰 DOM。这靠的是按行订阅加结构共享。"

**〔动作〕** 保持 2000/s，再滚动列表。

> "我一边保持每秒两千条更新、一边滚动——你看，还是满帧。这基本就是这道题最想看的东西了。"

## ④ 筛选 + 搜索，还在翻动中（约 70 秒）——"正确性"

**〔动作〕** 速率保持高位。点 **Failed** 状态 chip。

> "现在数据还在疯狂翻动，我点一下 Failed 这个筛选——列表立刻只剩失败的单子，右边这个数字（X of 10,000）也在实时跳。
>
> 关键是：底下的数据一直在变、失败的单子不断有进有出，但**你看到的每一行始终都是 Failed**，不会串。"

**〔动作〕** 在搜索框输入 `pharma`。

> "我再叠一个搜索，搜 pharma——现在是'状态是 Failed'**并且**'名字里有 pharma'的交集，还是对的。"

**〔动作〕** 稍作停顿，解释洞察。

> "这里有个我觉得挺关键的观察：更新只会改状态，永远不会改单号和客户名。所以搜索的结果其实不受数据翻动影响，只有状态筛选需要跟着重算——想清楚这一点，'边翻动边筛选还要正确'这个难点就被缩小了。"

**〔动作〕** 点 **Clear** 复位。

## ⑤ 收尾（约 40 秒）——质量与工程

**〔动作〕** 另开一个终端，敲 `npm test`。

> "最后花十秒钟——我写了 26 个单测，但只测真正难的部分：批处理、翻动下的筛选、feed 引擎、CSV 解析；没有去堆那些 UI 快照测试，题目也明说不需要。
>
> 整个架构是三层：feed、共享协议、前端；边界很清楚，其实也就是团队里天然的分工线。每个技术选型为什么这么选，我都在 DECISIONS 里写了取舍。"

## ⑥ 扩容论证（白话口播版，演示完主动引，别等冷场）

**〔过渡，一句话把节奏接过来〕**

> "刚才这个是 1 万行、单看板、在线的状态。需求里写明终面要"针对扩容场景论证架构可行性"。所以我这套架构其实是**刻意留了升级空间**的

**〔总纲，先给一句让他们抓住主线〕**

> "先说个总的思路：这三种情况我**都不用推倒重来**。我这三层之间是靠一份‘协议’隔开的，就像预留好的接口。放大的时候，我只在对应的那一层**换个更强的实现**，协议不动、别的层也不动。所以扩容对我来说不是‘能不能做’，而是‘到什么时候、该换哪一层、代价多大’。"

**① 数据量再大 10 倍（十万行）**

> "先说结论：**画面这一层根本不用动**——因为不管数据是一万行还是十万行，页面上真正渲染的永远只有几十行，虚拟化早就把它摁住了。
>
> 真正的压力会**按顺序**往后走：第一个顶不住的是‘筛选和搜索’，因为它要扫全部数据、会卡住输入——那我就把筛选**挪到一个后台线程**去算，主线程只管画，输入就不卡了。要是数据再大、每次还嫌慢，我就不再每次扫全部，而是**建个索引，只动变化的那几条**。等到数据大到根本传不进浏览器，那就**把数据留在服务端，前端只问服务端要‘现在屏幕上能看到的这几百行’**。
>
> 关键是我**不会一上来就全上**——先拿数据说话，看到真卡了再上对应那一招。现在一万行，最简单的做法就是最好的做法。"

**② 多个看板同时运行**

> "这个分两头看。
>
> 一头是**用户这边**：一个人可能开好几个标签页看不同看板，现在是每个标签各连一条服务器。放大就让**一个浏览器只连一条**、几个标签共用，省掉重复连接和重复流量。
>
> 另一头是**服务器这边**：看板多了、用户多了，一台服务器扛不住就要多开几台。这里的关键是让每台服务器**不存自己的状态**，这样才能随时加机器、减机器；服务器之间再放一个**消息中转**负责把更新广播出去。而且我这个数据**天生就能拆**——按仓库、按单号切开，**一个看板就是一个频道**，互相不影响，谁也不拖累谁。"

**③ 线下仓库断网**

> "我会先反问一句：**这些仓库是只‘看’看板，还是断网时也要能‘改’？** 这俩的复杂度差很远。
>
> 如果只是看——那‘断网’其实就是‘和服务器的连接断了’。我在**本地存一份最近的数据**，一断网就先拿本地的顶上，界面上挂个‘正在重连、数据可能不是最新’的提示，人照样能看。**网一恢复，我不重新下载全部，只把断网这段时间漏掉的那几条补回来**就行。
>
> 如果断网时还要能改（比如仓库现场标记‘已送达’）——那才需要更重的一套：**改动先记在本地、排好队，等联网了再一条条补提交给服务器**，服务器说了算。这种冲突用‘谁的时间戳新听谁的’就够了，不用上那种很重的自动合并机制——那是给多人协作编辑文档用的，我们这个场景用不上。"

**〔收束，把姿态收回来〕**

> "所以我想说的是：这套架构不是‘现在能跑就完了’，而是**每一层都留好了往上顶的路**。我能说清每一步——为什么现在不做、什么信号出现时该做、以及代价有多大。"

> "我这边先演示到这，接下来可以按你们想看的方向继续加东西。"

## 现场扩展预案（面试官很可能让你临场改，选一个讲）

- **加一列**（比如 ETA / 目的地）：改 `shared/protocol.ts` 的类型，加到 `ShipmentRow` 和 CSS grid 的 `--cols`——顺便讲"协议先行、两端不漂移"。
- **状态计数**：在每个筛选 chip 上显示实时数量——讲"是每帧重算、还是在 store 里增量维护"的取舍。
- **列排序**：讲排序 + 虚拟化 + 翻动三者叠加，需要稳定排序键和防抖。
- **再往上拉速率**：拖到滑杆上限，讲 `MAX_RATE`、以及什么时候该把筛选挪到 Web Worker。

## 万一翻车（保持镇定，边修边说）

- 页面红色 `disconnected`：feed 没起来，看终端 `[feed]` 日志；顺带说一句"客户端本身会自动重连，feed 一恢复它就自愈"，然后重启 feed。
- 端口占用：`lsof -ti tcp:8080 tcp:5173 | xargs kill -9`，再 `npm run dev`。
- 想要更猛的流动：拖满滑杆，或重启时 `FEED_RATE=2000 npm run dev`。
