  三、逐决策点方案对比（调研支撑）

  ① Feed 传输：独立 WS 服务 / 浏览器内 Worker / SSE。WS 最贴近真实生产 feed，ws 库轻量，concurrently 同起即"单命令"，且本地 Node 进程不算"外部服务"（外部=Redis/云
  broker）。研究也指出 WS 适合双向、未来可扩展场景（SSE vs WS 实务）。

  ② 状态管理（本题胜负手）：调研明确——高频场景下 naive Zustand selector 会整表重渲染，需手动 useShallow；Valtio proxy 自动按行但 10k 有开销、难调试；Jotai
  原子化天然隔离但 1 万 atom 偏重（Zustand/Valtio/Jotai 对比）。而 React 官方高频推荐范式是 ref 缓冲 + rAF 刷新 + useSyncExternalStore 按行订阅（React
  高频更新最佳实践、useSyncExternalStore 官方）。自建极简外部 store + 按行订阅最能展示 Staff 级性能工程深度，也最好 defend。

  ③ 虚拟化库：TanStack Virtual（2026 主流、活跃、headless）/ react-window（轻量但已停更、仅定高）/ react-virtuoso（开箱快但封装重）。10k 行两者都能稳
  60fps，瓶颈在行渲染器而非库本身（虚拟化库对比 2026）。

  ④ 更新批处理：核心范式已锁定——WS delta 先进 useRef 缓冲，requestAnimationFrame 每帧刷一次，把任意数量消息塌缩成每帧 1 次 render；同 reference
  多次更新只留最新；值未变则跳过（rAF 批处理）。这是"每秒数百条仍 60fps"的物理基础。

  实时 feed 的传输/架构选哪种？（PDF 明确要求「be ready to defend it」）

❯ 1. 独立 WebSocket 服务 ✔
     Node ws 库单独进程。最贴近真实生产 feed，能 defend 传输/背压/重连；concurrently 同起=单命令，本地进程非「外部服务」；终面「多看板/10x」叙事最强。我的推荐。
  2. 浏览器内 Web Worker
     生成器跑在浏览器 Worker，零网络、真单进程、最简。但传输深度弱、scale 叙事差，Staff 角色显得回避难点。
  3. SSE 单向推送
     比 WS 简单、单向够用。但双向/未来交互扩展弱，实时物流场景 WS 更主流、更好讲。

高频更新的状态管理层用哪种？（本题性能胜负手，终面必被追问）

  1. 自建外部store+按行订阅
     极简 external store + useSyncExternalStore，每行独立订阅→改一行只重渲染一行，配 rAF 批处理。最能展示性能工程深度、可 defend 到底层原语。代码量略多。我的推荐。
❯ 2. Zustand ✔
     务实、生态成熟（底层也是 useSyncExternalStore）。但按行订阅较别扭，naive selector 会整表重渲染需手动 useShallow。
  3. Valtio
     proxy 自动按访问追踪，天然按行隔离。但 1 万 proxy 有开销、调试较难、可 defend 性弱于自建。
  4. Jotai
     原子化按行天然隔离。但 1 万 atom 偏重，更适合派生态复杂场景，非本题最优。

表格虚拟化库选哪个？

❯ 1. TanStack Virtual ✔
     2026 主流、活跃维护、headless 可控，配 Staff 级性能叙事最佳；学习曲线略陡但可控。我的推荐。
  2. react-window
     轻量、久经考验、API 极简。但已停止活跃维护、仅支持定高、偏老派。
  3. react-virtuoso
     开箱即用、上手最快。但封装较重、可控性与可 defend 性弱于 headless 方案。

4-6 小时 timebox 下的范围策略？

❯ 1. MVP扎实+1个亮点 ✔
     核心(虚拟化/rAF批处理/churn下筛选)做到无懈可击 + 性能 HUD 抓眼球点，其余进「下一步」清单。契合 PDF「未完成但有理有据 > 镀金」。我的推荐。
